## Overview

The platform implements a **4-tier memory architecture** inspired by human memory systems to provide contextually-aware legal assistance. Each memory type serves a specific purpose and retrieval strategy.

```
┌─────────────────────────────────────────────────────────────┐
│                     MEMORY HIERARCHY                        │
├─────────────────────────────────────────────────────────────┤
│  Immediate Memory   →  Last 10 messages (conversation)      │
│  Session Memory     →  Current case data (structured JSON)  │
│  Episodic Memory    →  Past conversations (semantic search) │
│  Semantic Memory    →  Legal knowledge base (RAG)           │
└─────────────────────────────────────────────────────────────┘
```

## Architecture

## 1. Immediate Memory

**Purpose:** Short-term conversation context (working memory)

**Storage:** Last 10 messages in chronological order

**Retrieval:** Direct read, no search

### Implementation

```python
# backend/src/application/services/memory_service.py:26
async def store_immediate_memory(self, case_id: int, content: str):
    """Almacena memoria inmediata (últimos mensajes)"""
    self.memory_repo.add_memory(case_id, "immediate", content)
    
    # Mantener solo últimos 10
    memories = self.db.query(Memory).filter(
        Memory.case_id == case_id,
        Memory.kind == "immediate"
    ).order_by(Memory.created_at.desc()).all()
    
    if len(memories) > 10:
        for mem in memories[10:]:
            self.db.delete(mem)
        self.db.commit()
```

### Usage Example

```python
# Store conversation turn
await memory_service.store_immediate_memory(
    case_id=123,
    content="Assistant: El trámite de divorcio es gratuito en la Defensoría."
)

# Retrieve recent conversation
immediate = await memory_service.retrieve_immediate_memory(case_id=123)
# Returns: [
#   "User: ¿Cuánto cuesta el divorcio?",
#   "Assistant: El trámite de divorcio es gratuito en la Defensoría.",
#   ...
# ]
```

**Key Characteristics:**
- Automatic pruning (keeps only 10 most recent)
- No embeddings (fast storage/retrieval)
- Sequential order preserved
- Cleared at conversation end

## 2. Session Memory

**Purpose:** Structured case data for current session

**Storage:** Key-value pairs as JSON objects

**Retrieval:** Direct lookup by key

### Implementation

```python
# backend/src/application/services/memory_service.py:41
async def store_session_memory(self, case_id: int, key: str, value: Any):
    """Almacena datos de sesión como JSON"""
    import json
    content = json.dumps({key: value})
    
    # Actualizar o crear
    existing = self.db.query(Memory).filter(
        Memory.case_id == case_id,
        Memory.kind == "session",
        Memory.content.like(f'%"{key}":%')
    ).first()
    
    if existing:
        data = json.loads(existing.content)
        data[key] = value
        existing.content = json.dumps(data)
        self.db.commit()
    else:
        self.memory_repo.add_memory(case_id, "session", content)
```

### Usage Example

```python
# Store case facts
await memory_service.store_session_memory(case_id, "tipo_tramite", "divorcio")
await memory_service.store_session_memory(case_id, "tiene_hijos", True)
await memory_service.store_session_memory(case_id, "hijos_menores", 2)
await memory_service.store_session_memory(case_id, "regimen_matrimonial", "gananciales")

# Retrieve all session data
session_data = await memory_service.retrieve_session_data(case_id)
# Returns: {
#   "tipo_tramite": "divorcio",
#   "tiene_hijos": true,
#   "hijos_menores": 2,
#   "regimen_matrimonial": "gananciales"
# }
```

**Key Characteristics:**
- Structured data (not raw text)
- Upsert behavior (updates existing keys)
- Persists entire session duration
- Used for form data, case attributes, user preferences

## 3. Episodic Memory

**Purpose:** Searchable history of past conversations

**Storage:** Summarized conversation episodes with embeddings

**Retrieval:** Semantic search using vector similarity

### Implementation

```python
# backend/src/application/services/memory_service.py:61
async def store_episodic_memory(self, case_id: int, summary: str):
    """Almacena resumen episódico con embedding para búsqueda semántica"""
    # Generar embedding
    embeddings = await self.llm.embed([summary])
    embedding = None
    if embeddings and embeddings[0]:
        emb = embeddings[0]
        # Asegurar vector 1D
        if isinstance(emb, list) and len(emb) > 0 and isinstance(emb[0], list):
            emb = emb[0]
        embedding = emb
    
    mem = Memory(
        case_id=case_id,
        kind="episodic",
        content=summary,
        embedding=embedding
    )
    self.db.add(mem)
    self.db.commit()
```

### Semantic Search

Uses **pgvector** for efficient similarity search:

```python
# backend/src/application/services/memory_service.py:110
async def search_episodic_memory(self, case_id: int, query: str, limit: int = 5):
    """Búsqueda semántica en memoria episódica usando embeddings"""
    # Generate query embedding
    embeddings = await self.llm.embed([query])
    query_embedding = embeddings[0]
    
    # Vector similarity search (PostgreSQL + pgvector)
    embedding_str = str(query_embedding).replace(' ', '')
    
    result = self.db.execute(
        text("""
            SELECT id, content,
                   embedding <-> CAST(:query_embedding AS vector) AS distance
            FROM memories
            WHERE case_id = :case_id AND kind = 'episodic'
            ORDER BY distance
            LIMIT :limit
        """),
        {
            "query_embedding": embedding_str,
            "case_id": case_id,
            "limit": limit,
        },
    ).fetchall()
    
    return [{"id": r[0], "content": r[1], "distance": r[2]} for r in result]
```

### Usage Example

```python
# Store conversation summary at session end
summary = await memory_service.summarize_conversation(case_id)
await memory_service.store_episodic_memory(
    case_id=123,
    summary="Usuario consultó sobre costos de divorcio (gratuito). Tiene 2 hijos menores. Necesita acta de matrimonio."
)

# Search relevant past episodes
episodes = await memory_service.search_episodic_memory(
    case_id=123,
    query="documentos necesarios para divorcio",
    limit=3
)
# Returns: [
#   {
#     "id": 456,
#     "content": "Usuario consultó sobre costos de divorcio...",
#     "distance": 0.23  # Low distance = high similarity
#   },
#   ...
# ]
```

**Key Characteristics:**
- Summarized (not full transcripts)
- Vector embeddings for semantic search
- Cross-session retrieval
- Sorted by relevance (not chronology)

## 4. Semantic Memory

**Purpose:** Domain knowledge base (legal procedures, laws, FAQs)

**Storage:** Pre-embedded legal documents in `semantic_knowledge` table

**Retrieval:** RAG (Retrieval-Augmented Generation) via vector search

### Implementation

```python
# backend/src/application/services/memory_service.py:168
async def search_semantic_knowledge(self, query: str, limit: int = 3):
    """Búsqueda en base de conocimiento legal semántico"""
    embeddings = await self.llm.embed([query])
    query_embedding = embeddings[0]
    
    embedding_str = str(query_embedding).replace(' ', '')
    
    result = self.db.execute(
        text("""
            SELECT id, title, content,
                   embedding <-> CAST(:query_embedding AS vector) AS distance
            FROM semantic_knowledge
            ORDER BY distance
            LIMIT :limit
        """),
        {
            "query_embedding": embedding_str,
            "limit": limit,
        },
    ).fetchall()
    
    return [
        {"id": row[0], "title": row[1], "content": row[2], "distance": row[3]}
        for row in result
    ]
```

### Usage Example

```python
# Search legal knowledge base
knowledge = await memory_service.search_semantic_knowledge(
    query="requisitos para divorcio con hijos menores",
    limit=2
)
# Returns: [
#   {
#     "id": 12,
#     "title": "Divorcio con Hijos Menores",
#     "content": "Cuando hay hijos menores, se debe presentar: 1) Plan de parentalidad...",
#     "distance": 0.18
#   },
#   {
#     "id": 34,
#     "title": "Documentación Requerida",
#     "content": "DNI de ambos cónyuges, Acta de matrimonio actualizada...",
#     "distance": 0.31
#   }
# ]
```

**Key Characteristics:**
- Global scope (not case-specific)
- Pre-populated by administrators
- High-quality curated content
- Updated independently of conversations

## Unified Context Building

Combines all 4 memory types for LLM prompts:

```python
# backend/src/application/services/memory_service.py:218
async def build_context_for_llm(self, case_id: int, current_question: str) -> str:
    """Construye contexto completo para el LLM"""
    context_parts = []
    
    # 1. Immediate memory (last 10 messages)
    immediate = await self.retrieve_immediate_memory(case_id)
    if immediate:
        context_parts.append("## Conversación reciente:\n" + "\n".join(immediate))
    
    # 2. Session data (structured case facts)
    session_data = await self.retrieve_session_data(case_id)
    if session_data:
        context_parts.append(f"## Datos del caso:\n{json.dumps(session_data, indent=2)}")
    
    # 3. Episodic memory (relevant past conversations)
    episodic = await self.search_episodic_memory(case_id, current_question, limit=3)
    if episodic:
        episodes = "\n".join([f"- {ep['content']}" for ep in episodic])
        context_parts.append(f"## Conversaciones anteriores relevantes:\n{episodes}")
    
    # 4. Semantic knowledge (legal documentation)
    knowledge = await self.search_semantic_knowledge(current_question, limit=2)
    if knowledge:
        docs = "\n\n".join([f"**{k['title']}**\n{k['content']}" for k in knowledge])
        context_parts.append(f"## Conocimiento legal aplicable:\n{docs}")
    
    return "\n\n".join(context_parts)
```

## Retrieval Strategies

| Memory Type | Retrieval Method | Latency | Use Case |
|-------------|------------------|---------|----------|
| **Immediate** | Direct read (ORDER BY timestamp) | &lt;10ms | Conversation continuity |
| **Session** | JSON key lookup | &lt;10ms | Form autofill, case facts |
| **Episodic** | Vector similarity (pgvector) | ~50ms | "Have we discussed X before?" |
| **Semantic** | Vector similarity (pgvector) | ~50ms | Legal knowledge retrieval |

## Performance Optimization

### Vector Indexing

Create pgvector indexes for fast similarity search:

```sql
-- Episodic memory index
CREATE INDEX idx_memories_embedding ON memories 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

-- Semantic knowledge index
CREATE INDEX idx_knowledge_embedding ON semantic_knowledge 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 50);
```

### Embedding Caching

Query embeddings are cached for repeated searches:

```python
# Cache query embeddings in Redis
cache_key = f"embedding:{hashlib.md5(query.encode()).hexdigest()}"
cached = await redis.get(cache_key)

if cached:
    query_embedding = json.loads(cached)
else:
    query_embedding = await llm.embed([query])[0]
    await redis.setex(cache_key, 3600, json.dumps(query_embedding))
```

## Best Practices

1. **Immediate Memory** - Store every user/assistant turn for context
2. **Session Memory** - Update incrementally as data is collected
3. **Episodic Memory** - Summarize at natural breakpoints (session end, topic change)
4. **Semantic Memory** - Pre-populate with curated legal content
5. **Context Size** - Monitor token usage when combining all 4 types
6. **Privacy** - Episodic memory should anonymize sensitive PII
