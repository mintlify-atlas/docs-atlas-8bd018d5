## Overview

The Defensoría Civil divorce platform implements **Clean Architecture** with strict dependency rules. All dependencies point inward toward the domain, ensuring the core business logic remains independent of external frameworks and tools.

## Four-Layer Architecture

The system follows a four-layer architecture:

**1. Domain Layer (Core)** - Pure business entities and rules
- Entity models (Case, User, Message, Memory)
- No external dependencies
- Located in `core/`

**2. Application Layer** - Use cases and application logic
- Use Cases (ProcessIncomingMessageUseCase, AuthenticateUserUseCase)
- Service interfaces and DTOs
- Business orchestration
- Located in `application/`

**3. Infrastructure Layer** - External integrations and implementations
- Database repositories (PostgreSQL + pgvector)
- LLM clients (Gemini, Ollama)
- WhatsApp service (WAHA integration)
- OCR services
- Located in `infrastructure/`

**4. Presentation Layer** - API and user interfaces
- FastAPI REST endpoints
- Request/response models
- HTTP middleware
- Located in `presentation/`

## Presentation Layer

### FastAPI Application

The FastAPI app provides RESTful API endpoints with automatic OpenAPI documentation. Security middleware includes CORS, rate limiting (30 req/min per IP), security headers (HSTS, CSP), and structured logging.

Router groups:
- Health check endpoint
- Authentication (JWT tokens)
- WhatsApp webhook
- Case management
- System metrics
- User administration

### Middleware Stack

Three security middlewares process every request:

- **RequestLoggingMiddleware** - Structured logging with method, path, status, and duration
- **SecurityHeadersMiddleware** - Security headers (X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, CSP)
- **RateLimitMiddleware** - Redis-backed distributed rate limiting

## Application Layer

### Use Cases

**ProcessIncomingMessageUseCase** - Main message processing orchestrator:
- Retrieves or creates case by phone number
- Processes media (OCR for DNI/marriage certificates)
- Manages state machine phases (28+ phases from inicio to documentacion)
- Validates responses with hallucination detection
- Stores messages and updates memory

**AuthenticateUserUseCase** - JWT authentication:
- Validates credentials with bcrypt
- Generates JWT tokens with user role
- Enforces active account status

**IngestLegalDocumentUseCase** - Legal knowledge ingestion:
- Chunks documents for embedding
- Generates vector embeddings
- Stores in semantic knowledge base

### Services

**MemoryService** - Four-tier memory management:
- Immediate memory (last 10 messages)
- Session memory (current case data as JSON)
- Episodic memory (past conversations with semantic search)
- Semantic memory (legal knowledge base)

**HallucinationDetectionService** - Response validation:
- Rule-based detection (pattern matching for hallucinations)
- LLM-based semantic validation
- Confidence scoring with 95% threshold

**ConversationEngine** - Legacy state machine handler (being migrated to ProcessIncomingMessageUseCase)

## Infrastructure Layer

### Database (PostgreSQL + pgvector)

**Models:**
- Case - Divorce case with personal data, spouse info, children, economic profile
- User - Operator/admin accounts with roles
- Message - Conversation history
- Memory - Four memory types with vector embeddings
- SemanticKnowledge - Legal knowledge base
- SupportDocument - Additional documents (employment proof, etc.)

**pgvector Extension:**
- Stores embeddings for episodic and semantic memory
- Enables semantic search with cosine similarity
- Indexed with IVFFlat for performance

### AI Router

**LLMRouter** - Multi-provider LLM management:
- Primary: Ollama Cloud (minimax-m2, deepseek-v3.1, qwen3-vl, glm-4.6)
- Fallback: Ollama Local
- Final fallback: Google Gemini
- Task-specific model selection (chat, reasoning, vision, hallucination detection, embeddings)

### WhatsApp Integration (WAHA)

- Receives messages via webhook
- Sends responses (text, buttons, lists)
- Handles media (images, documents)
- Message deduplication with Redis

### OCR Service

**MultiProviderOCRService:**
- Gemini Vision for DNI and marriage certificate extraction
- Structured data extraction with validation
- Confidence scoring per document type
- Cross-validation against declared data

## Data Flow

End-to-end message processing:

<Steps>
  <Step title="WhatsApp Webhook">
    User sends message via WhatsApp, WAHA forwards to webhook endpoint
  </Step>

  <Step title="Request Processing">
    FastAPI receives request, applies middleware (logging, security, rate limiting)
  </Step>

  <Step title="Safety Check">
    SafetyLayer filters input for prompt injection and inappropriate content
  </Step>

  <Step title="Case Retrieval">
    CaseRepository retrieves or creates case by phone number
  </Step>

  <Step title="Memory Storage">
    Message stored in immediate memory (last 10 messages)
  </Step>

  <Step title="Phase Handling">
    State machine processes message based on current phase (tipo_divorcio, apellido, etc.)
  </Step>

  <Step title="LLM Generation">
    If needed, LLMRouter generates response with context from MemoryService
  </Step>

  <Step title="Hallucination Check">
    HallucinationDetectionService validates response accuracy (rule-based + LLM-based)
  </Step>

  <Step title="Response Storage">
    Assistant response stored in database and memory
  </Step>

  <Step title="WhatsApp Reply">
    WAHAWhatsAppService sends response (text, buttons, or lists)
  </Step>
</Steps>

## Deployment Architecture

Docker Compose orchestrates 5 services:

- **api** - FastAPI backend (port 8000)
- **worker** - Celery worker for async tasks
- **db** - PostgreSQL 14 with pgvector
- **redis** - Redis 7 for cache and rate limiting
- **dashboard** - React frontend (port 5173)

External integration: **WAHA** for WhatsApp (port 3000)

## Security Architecture

**Authentication & Authorization:**
- JWT tokens with 24-hour expiration
- Role-based access (operator vs admin)
- bcrypt password hashing

**API Security:**
- Rate limiting (30/min per IP, 100/min per user)
- Security headers (HSTS, CSP, X-Frame-Options)
- CORS configuration for trusted origins
- Input validation with Pydantic models
- Prompt injection protection

**Data Protection:**
- Environment variable secrets
- Database connection encryption
- Structured logging (no sensitive data in logs)

## Performance Considerations

**Asynchronous I/O:**
- FastAPI async/await throughout
- Async database operations with SQLAlchemy 2.0
- Async LLM calls with httpx

**Caching:**
- Redis for rate limit counters
- Session memory cached in Redis (planned)
- LLM response caching (planned)

**Database Optimization:**
- pgvector indexes for semantic search
- Connection pooling
- Selective eager loading

**LLM Latency:**
- Target < 5 seconds (95th percentile)
- Multiple provider fallbacks
- Prompt optimization

## Observability

**Structured Logging:**
- All requests logged with structlog
- Log levels: debug, info, warning, error
- Contextual fields (case_id, phone, phase)

**Tracing:**
- OpenTelemetry instrumentation
- Distributed tracing across services
- Span attributes for debugging

**Metrics:**
- Total cases
- Cases by status and type
- Daily timeline
- Success rates (planned)

**Health Checks:**
- Liveness endpoint
- Database connectivity check
